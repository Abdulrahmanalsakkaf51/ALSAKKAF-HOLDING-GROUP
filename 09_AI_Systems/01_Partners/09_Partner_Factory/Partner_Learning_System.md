# 1. Purpose

This document defines how Partners improve over time: what they remember, what they learn from, how lessons become proposals, and how proposals become approved changes.

Partners must be able to get better without ever silently changing their own identity, permissions, or skills.

---

# 2. Local Memory

Local memory is the general institutional knowledge a Partner may draw on: approved AOS documents, registers, and prior lessons learned, accessed through The Librarian's indexing.

Local memory is read-only for the Partner. A Partner cannot write new "official" knowledge into local memory without the same approval discipline used elsewhere in AOS.

---

# 3. Role Memory

Role memory is the knowledge specific to a Partner's purpose: prior tasks it has handled, patterns in the requests it receives, and context relevant only to its role.

Role memory helps a Partner respond consistently within its purpose but does not expand its purpose. A Partner encountering a task outside its role memory's normal pattern still checks it against its defined purpose and permission level, not against convenience.

---

# 4. Task Outcomes

Every task a Partner completes should be logged with:

| Field | Description |
|-------|-------------|
| Task description | What was asked |
| Partner used | Which Partner handled it |
| Model tier used | Per the Partner Routing and Cost Control Model |
| Outcome | Success, partial success, failure, escalated to Founder |
| Cost | AI cost incurred, if any |

Task outcomes are the raw material for lessons learned.

---

# 5. Lessons Learned

Recurring patterns in task outcomes are turned into lessons learned entries, following the same discipline used in prior AOS projects (for example, PRJLESSON-001 through PRJLESSON-005).

A lesson is not automatically a change. A lesson is an observation that may justify a future change.

---

# 6. Feedback Loops

```text
Task completed
    ↓
Outcome logged
    ↓
Patterns reviewed periodically (by Atlas, for the Founder)
    ↓
Lesson learned drafted if a pattern is significant
    ↓
Lesson reviewed by Founder
    ↓
Improvement proposal drafted if the lesson justifies a change
```

Feedback loops apply to individual Partners and to the Factory system itself.

---

# 7. Improvement Proposals

An improvement proposal is a specific, documented suggestion to change a Partner's skills, prompt, permission level, or routing behavior.

Every improvement proposal must state:

| Field | Description |
|-------|-------------|
| Partner affected | Which Partner this changes |
| Current behavior | What the Partner does today |
| Proposed change | What would change |
| Evidence | Which task outcomes or lessons justify the change |
| Risk | What could go wrong if approved |

Improvement proposals are prepared by Atlas, not self-generated by the Partner being changed.

---

# 8. Skill Versioning

When an improvement proposal is approved, the affected Partner's profile, skill list, or prompt is updated as a new version, with a changelog entry recording what changed and why.

Prior versions remain in the record. Version history is never erased.

---

# 9. Approved Learning Versus Uncontrolled Self-Modification

| Approved Learning | Uncontrolled Self-Modification |
|--------------------|--------------------------------|
| Task outcomes are logged and reviewed | A Partner changes its own prompt or skills without review |
| Lessons are documented and reviewed by the Founder | A Partner infers new permissions for itself |
| Improvement proposals are explicit and evidence-based | A Partner silently expands its purpose over time |
| Changes create a new documented version | Changes happen with no record |
| Founder approves before activation | Change takes effect without approval |

Uncontrolled self-modification is never permitted at any Partner Authority Level, including Level 3.

---

# 10. How Partners Become Better Over Time

```text
Partner operates within approved purpose and permission level
    ↓
Task outcomes accumulate
    ↓
Patterns become lessons
    ↓
Lessons become improvement proposals
    ↓
Founder approves or rejects
    ↓
Approved changes become a new Partner version
    ↓
Partner continues, now slightly better, with full history intact
```

---

# 11. How The Librarian Indexes Learning

The Librarian indexes lessons learned entries, improvement proposals, and version changelogs for every Partner, so that patterns across Partners (not just within one Partner) become discoverable over time.

This allows AOS to notice, for example, that several Partners independently surfaced the same gap, which is itself valuable institutional knowledge.

---

# 12. How Atlas Reviews Learning for CEO Dashboards

Atlas periodically reviews accumulated task outcomes, lessons, and improvement proposals across all active Partners, and prepares a summarized view suitable for CEO-level dashboard reporting: how many Partners are active, what they are learning, what changes are pending Founder review, and where cost or performance concerns are emerging.

Atlas prepares this summary. Atlas does not approve the underlying changes.
