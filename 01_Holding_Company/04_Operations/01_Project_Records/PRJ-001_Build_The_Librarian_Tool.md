# 1. Purpose

The purpose of this project is to turn The Librarian from a prompt-based Partner into a more practical working knowledge tool.

The Librarian is already active as a prompt-based Partner.

This project will improve The Librarian so it can become easier to use, test, update, and later connect to the ALSAKKAF HOLDING GROUP repository.

---

# 2. Problem Statement

ALSAKKAF HOLDING GROUP now has many important documents, including:

- Digital DNA
- Knowledge Register
- Architecture Decision Log
- ADR files
- AOS University lessons
- Partner Operating Model
- Partner Registry
- Partner Workforce Architecture
- The Librarian profile
- The Librarian prompt
- The Librarian test log
- Project Operating Model

As the institution grows, it will become harder to remember where everything is stored.

The current Librarian works inside ChatGPT, but it is not yet a reusable tool connected to the project structure.

A practical Librarian Tool is needed to help find institutional knowledge faster and more reliably.

---

# 3. Project Goal

The goal of PRJ-001 is to create the first practical version of The Librarian Tool.

This project should begin simple.

The first goal is not to build a complex AI application.

The first goal is to create a clean working foundation that can later become software.

---

# 4. Scope

This project includes:

- Creating a document map for The Librarian
- Creating a reusable Librarian workflow
- Creating a simple testing process
- Preparing the knowledge sources
- Preparing future software requirements
- Defining the first build stages
- Identifying what Claude may help code later

---

# 5. Out of Scope

This project does not yet include:

- A full web application
- A paid SaaS platform
- Autonomous AI agents
- Direct GitHub API integration
- Customer-facing tools
- Multi-user permissions
- Legal or financial automation
- Autonomous decision-making

These may come later after the foundation works.

# 9. Risks

| Risk | Mitigation |
|------|------------|
| Building too complex too early | Start with a manual workflow and document map |
| Losing track of documents | Create a source map |
| Prompt gives unsupported answers | Use test log and source priority rules |
| Tool becomes confusing | Keep v1 simple |
| Coding before requirements are clear | Document requirements before Claude coding |
| Treating AI output as final truth | Keep human approval required |

---

# 10. Project Stages

## Stage 1 — Manual Foundation

Create:

- Document map
- Librarian workflow
- Source priority map
- Reusable prompt package

## Stage 2 — Tool Requirements

Define:

- What the tool should do
- What files it should read
- What outputs it should produce
- What it should not do
- How it should be tested

## Stage 3 — Simple Prototype

Possible prototype options:

- Command-line tool
- Local Python script
- Simple document search helper
- Markdown index reader

## Stage 4 — Repository-Connected Version

Later, connect The Librarian to the GitHub repository or local folder structure.

## Stage 5 — Advanced Partner Version

Later, The Librarian may become part of a larger Partner dashboard.
