# 1. Purpose

The Project Operating Model defines how ALSAKKAF HOLDING GROUP creates, approves, manages, tracks, completes, and learns from projects.

AOS does not treat ideas as projects automatically.

An idea becomes a project only when it has:

- A clear problem
- A clear purpose
- A defined owner
- Expected outcomes
- Required resources
- A decision to proceed
- A documented project record

This model protects ALSAKKAF HOLDING GROUP from random execution.

---

# 2. What Is a Project?

A project is a structured effort created to solve a defined problem or produce a specific outcome.

A project has:

- A name
- A purpose
- An owner
- A scope
- A timeline
- Deliverables
- Risks
- Decisions
- Progress records
- Lessons learned

A project is not just an idea.

A project is an approved path toward execution.

---

# 3. Project Philosophy

AOS uses projects to turn ideas into real outcomes.

Projects help ALSAKKAF HOLDING GROUP:

- Stay focused
- Avoid random work
- Track progress
- Use Partners responsibly
- Capture knowledge
- Learn from success and failure
- Build repeatable systems
- Move toward revenue and impact

---

# 4. Project Lifecycle

Projects follow the AOS Lifecycle.

```text
Dream
  ↓
Question
  ↓
Problem
  ↓
Research
  ↓
Architecture
  ↓
Decision
  ↓
Project
  ↓
Prototype
  ↓
Validation
  ↓
Implementation
  ↓
Solution
  ↓
Feedback
  ↓
Knowledge
  ↓
AOS Improves
```

A project should not begin before the problem is clear.

# 5. Project Status Definitions

| Status | Meaning |
|--------|---------|
| Idea | Not yet structured |
| Proposed | Project has been suggested |
| Researching | Information is being gathered |
| Designed | Project structure is documented |
| Approved | Project is approved to begin |
| Active | Project is currently being executed |
| Paused | Project is temporarily stopped |
| Completed | Project deliverables are finished |
| Archived | Project is closed and stored |
| Rejected | Project was reviewed and not approved |
| Cancelled | Project began but was stopped |

---

# 6. Project Types

ALSAKKAF HOLDING GROUP may create different types of projects.

| Project Type | Purpose |
|-------------|---------|
| Foundation Project | Builds AOS, governance, systems, or institutional structure |
| Partner Project | Designs, tests, or improves a Partner |
| Technology Project | Builds software, automation, tools, or integrations |
| Business Project | Tests or launches a revenue-generating activity |
| Research Project | Studies a market, problem, opportunity, or industry |
| Operations Project | Creates workflows, SOPs, processes, or internal systems |
| Marketing Project | Builds campaigns, content, brand assets, or channels |
| Learning Project | Creates AOS University lessons or training material |
| Improvement Project | Improves existing systems, documents, Partners, or processes |

---

# 7. Project Requirements

Before a project becomes Active, it must have:

1. Project ID
2. Project name
3. Project type
4. Project owner
5. Problem statement
6. Purpose
7. Scope
8. Deliverables
9. Success criteria
10. Required Partners
11. Risks
12. Timeline
13. Approval status
14. Project record location

---

# 8. Project ID System

Projects should use a clear ID system.

Examples:

| Project ID | Meaning |
|-----------|---------|
| PRJ-001 | First official project |
| PRJ-002 | Second official project |
| PRJ-003 | Third official project |

Project IDs should not be reused.

Even cancelled or rejected projects should keep their IDs for institutional memory.

---

# 9. Project Record

Every official project should have a project record.

A project record should include:

- Project information
- Problem statement
- Purpose
- Scope
- Deliverables
- Timeline
- Partners involved
- Decisions
- Progress updates
- Risks
- Lessons learned
- Final outcome

The project record becomes part of institutional knowledge.

---

# 10. Role of the Founder / CEO

The Founder / CEO is responsible for:

- Approving major projects
- Setting direction
- Prioritizing work
- Reviewing risks
- Approving resources
- Making final decisions
- Ensuring alignment with the Constitution and Digital DNA

Partners may support the Founder, but they do not replace the Founder.

---

# 11. Role of Partners in Projects

Partners may support projects by:

- Researching information
- Preparing plans
- Drafting documents
- Creating checklists
- Summarizing progress
- Identifying risks
- Supporting decisions
- Preparing reports
- Capturing lessons learned

Partners may assist, recommend, and prepare.

They may not approve major decisions alone.

---

# 12. The Librarian and Projects

The Librarian supports projects by helping users find:

- Relevant AOS knowledge
- Related ADRs
- Previous project records
- Applicable templates
- Related lessons
- Partner documents
- Missing documentation

The Librarian helps projects start with existing knowledge instead of starting from zero.

---

# 13. Project Approval

A project may be approved only when:

1. The problem is clear.
2. The project has a defined purpose.
3. The scope is realistic.
4. The expected outcome is useful.
5. Risks are understood.
6. The required resources are reasonable.
7. The project aligns with AOS and the Constitution.

---

# 14. Project Reporting

Active projects should produce regular updates.

A project update may include:

- Current status
- Completed work
- Pending work
- Blockers
- Risks
- Decisions needed
- Next actions
- Lessons learned

Reporting keeps the Founder / CEO informed.

---

# 15. Project Completion

A project is completed when:

- Deliverables are finished
- Results are reviewed
- Lessons are captured
- Knowledge is documented
- Related records are updated
- The project is closed or moved to the next stage

A project is not truly complete until its knowledge returns to AOS.

---

# 16. Project Archive

Completed, cancelled, rejected, and paused projects should not disappear.

They should be archived when appropriate.

Archived projects help the institution remember:

- What was attempted
- What worked
- What failed
- Why decisions were made
- What should be improved next time

---

# 17. First Recommended Project

The first official project should be:

## PRJ-001 — Build The Librarian Tool

This project will turn The Librarian from a prompt-based Partner into a more practical working tool.

The project may begin with:

- A manual workflow
- A reusable prompt
- A document map
- A simple command-line tool
- A repository-connected assistant later

The purpose is to make institutional knowledge easier to search and use.

---

# 18. Project Operating Principle

> A project is not finished when the task is done.
>
> A project is finished when the institution has learned from it.

---

# 19. Project Operating Rule

No official project should begin inside ALSAKKAF HOLDING GROUP until its purpose, owner, scope, deliverables, success criteria, risks, and relationship to AOS are documented.
